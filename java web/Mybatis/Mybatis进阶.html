也有其他的方法Mybatis提供了2种参数占位符

  • #{}：执行SQL代码时，会将#{}占位符替换为?，底层使用的是preparedStatement

  • ${}：拼接SQL，拼接为字符串格式，底层使用的是Statement，会存在SQL注入问题，看如下代码

    <mapper namespace="com.zq.mapper.IUserMapper">

        select *
        from tb_user
        where name like '%${name}%';
    </select>
</mapper>

    上面的SQL代码的编译结果为

    select * from tb_user where name like '%张%';
    1

    如果用户在搜索的时候输入以下内容 %张%' or '1' = '1，在执行搜索方法的时候

    List<User> userList = mapper.getUserByName("%张%' or '1' = '1");
    1

    出现的结果是

    User{id=1, name='张三', age=18, gradeClass='大一', schoolName='武汉大学', address='武汉'}
User{id=2, name='李四', age=26, gradeClass='大三', schoolName='湖北大学', address='武汉'}
User{id=3, name='王五', age=23, gradeClass='大二', schoolName='长江大学', address='荆州'}
    1
    2
    3

    因此使用${}来作为参数占位符是不安全的，建议使用#{}。

    #{}与${}区别：

    1. #{}是预编译处理，${}是字符串替换
    2. Mybatis在处理#{}时，会将sql中的#{}替换为?号，调用PreparedStatement的set方法来赋值，Mybatis在处理${}时，Mybatis会把${}替换成变量的值。
    3. 使用#{}可以有效的防止SQL注入，提高系统安全性。原因在于：预编译机制。预编译完成之后，SQL的结构已经固定，即便用户输入非法参数，也不会对SQL的结构产生影响，从而避免了潜在的安全风险。

# 多条件查询(动态SQL)

情景分析：有的时候用户在搜索的时候，可能搜索多个字段，例如上面在根据用户名称和大学名称和年级等字段进行模糊查询的时候可能存在多个查询条件。

# 需求：(动态SQL)

根据用户名称和年级和学校名称进行模糊查询

对于多个参数的条件查询有多种实现方式，接口方法的3种实现

  1. 将多个参数封装成一个 实体对象

  2. 将多个参数封装到map集合中

    List<User> getUserConditions3(@Param("name") String name, @Param("gradeClass") String clasName, @Param("schoolName") String schoolName);
    1

        <choose>
            <when test="schoolName != null and schoolName != ''">

# 删除多条数据

情景分析：用户在清空购物车的时候实际上就是删除多条数据，用户在勾选要删除的数据的时候，最后点击删除，删除所有勾选的数据。

需求：根据多个用户的id，删除user表中的多条数据，删除下面3条数据

id name age grade_class school_name address
6 甲 11 初一 小学 湖北
7 乙 12 初二 小学 湖北
8 丙 15 高一 高中 湖北

解决办法：使用foreach标签

接口中的方法为

int deleteUserById2(int[] ids);
1

SQL语句如下

<delete id="deleteUserById2">
    delete
    from tb_user
    where id in
    <foreach collection="array" separator="," open="(" close=")" item="id">
        #{id}
    </foreach>
</delete>

关于foreach标签中collection的值

  • 如果方法传入的是单参数且参数类型是一个 List 的时候，collection 属性值默认为 list

  • 如果方法传入的是单参数且参数类型是一个 array 数组的时候，collection 的属性值默认为 array

  • 如果方法中使用的是注解来设置参数数组的名称，那么foreach标签中collection属性的值也要和注解的值一样

    int deleteUserById3(@Param("ids") int[] ids);

    #{id}

# 注解实现增删改查

注解是用来替换映射配置文件方式配置的，所以使用了注解，就不需要再映射配置文件中书写对应的 statement

Mybatis 针对 CURD 操作都提供了对应的注解，已经做到见名知意。如下：

  • 查询 ：@Select
  • 添加 ：@Insert
  • 修改 ：@Update
  • 删除 ：@Delete
  • 结果集映射：@ResultMap，这个注解为 @Select 指定 XML 映射中 <resultMap> 元素的 id。复用已在 XML 中定义的 ResultMap。
  • 设置可选配置项(例如返回自增的注解id)：@Options。

@Insert("insert into tb_user values (null,#{name},#{age},#{gradeClass},#{schoolName},#{address})")

# 动态增删改查

Mybatis 3.x版本提供了以下4个CRUD的高级注解

  1. @SelectProvider：用于构建动态查询SQL
  2. @InsertProvider：用于构建动态新增SQL
  3. @UpdateProvider：用于构建动态更新SQL
  4. @DeleteProvider：用于构建动态删除SQL

# 关于注解和xml映射语句

官方文档中的解释：

​ 使用注解来映射简单语句会使代码显得更加简洁，但对于稍微复杂一点的语句，Java 注解不仅力不从心，还会让本就复杂的 SQL 语句更加混乱不堪。 因此，如果你需要做一些很复杂的操作，最好用 XML 来映射语句。选择何种方式来配置映射，以及是否应该要统一映射语句定义的形式，完全取决于你和你的团队。 换句话说，永远不要拘泥于一种方式，你可以很轻松地在基于注解和 XML 的语句映射方式间自由移植和切换。
