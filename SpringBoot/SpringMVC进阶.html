# SpringMVC进阶

# RESTful风格

RESTful 是URL路径的一种设计风格。需遵循以下约定：

  1. 请求路径：请求参数尽量融合在路径中，避免通过？携带参数
  2. 请求方式：一般查询建议使用GET请求，添加使用POST请求，更新使用PUT请求，删除使用DELETE请求
  3. 数据格式：用 JSON 来定义数据的交互格式，也可以是其它格式

SpringMVC提供了 @GetMapping、@PostMapping、@PutMapping、@DeleteMapping四种不同的注解定义不同 请求方式映射路径。注意：只能写在方法上，不能写在类上

例如查询指定用户

@RestController
@RequestMapping("user")
public class UserController {

   //根据id查询
   @GetMapping("{id}")
   public String selectById(@PathVariable Integer id) {
      return "查询指定用户";
   }
}

# 参数校验

前端的参数校验主要是为了更好的用户体验，并不能保证其安全性，例如可以通过一些http请求工具绕过前端的参数校验，因此后端也需要自己做参数校验。

bindingResult和@Valid的使用

@Data
@AllArgsConstructor
@NoArgsConstructor
public class User {
    @NotBlank(message = "用户名不能为空！")
    private String name;

    @NotNull(message = "年龄不能为空")
    @Max(value = 150,message = "年龄不能大于150")
    @Min(value = 1, message = "年龄不能为0和负数")
    private Integer age;

    @NotNull(message = "手机号不能为空")
    @Pattern(regexp = "^1[356789]\\d{9}$",message = "手机号格式不合法")
    private String phone;
}

# 使用Spring自带的ResponseEntity

ResponseEntity是Spring提供的一个类，它内部封装了状态码，请求头，请求体等信息，可以根据自己的需要去设置状态码、请求体的信息。

@GetMapping("save1")
public ResponseEntity save1(@Valid User user,BindingResult result){
	if (result.hasErrors()) {
		Stream<String> stream = result.getFieldErrors().stream().map(item -> item.getDefaultMessage());
		List<String> errorList = stream.collect(Collectors.toList());
		return new ResponseEntity(errorList, HttpStatus.BAD_REQUEST);
	}else {
	    return ResponseEntity.ok("保存用户成功");
	}
}

# 自定义封装类 Result

响应状态码的不足，上一节说到，可以用状态码较为精确地描述此操作是成功失败、是客户端的问题、还是服务端的问题，是 最为通用的描述，但状态码毕竟有限，如果牵扯业务，就有些不够用了

返回的响应体中，可能会表示正常的数据，也有可能包含错误提示，若不统一，就增加了前端解析成本，因此一般应用开发时， 会定义一个 Result 类统一响应格式

@Data
@NoArgsConstructor
@AllArgsConstructor
public class Result {

    private Integer code;//200表示成功，400表示失败
    private String errorMsg;//失败信息描述
    private Object data;//页面要展示的数据

    //成功成功
    public static Result success(Object data) {
        Result result = new Result();
        result.setCode(200);
        result.setData(data);
        return result;
    }
    
    //失败信息
    public static Result error(String errorMsg) {
        Result result = new Result();
        result.setCode(400);
        result.setErrorMsg(errorMsg);
        return result;
    }
}

# 全局异常处理

在项目上线之后，往往会出现一些不可预料的异常信息，对于逻辑性或设计性问题，开发人员或者维护人员需要通过日志，查看异常信息并排除异常；而对于用户，则需要为其呈现出其可以理解的异常提示页面，让用户有一个良好的使用体验。所以异常的处理对于一个Web项目来说是非常重要的。Spring MVC提供了强大的异常处理机制。

# 全局异常处理

编写步骤：

  1. 编写一个类，在类上添加@RestControllerAdvice注解，标识该类为全局异常处理器。
  2. 编写一个方法，在方法上添加@ExceptionHandler注解，一个方法处理一种特定异常。
@RestControllerAdvice
@Slf4j
public class GlobalExceptionHandler {
   //  全局异常处理
   @ExceptionHandler(RuntimeException.class)
   public Result doExceptionHandler1(RuntimeException e) {
       log.error("错误信息为{}",e.getMessage());
      return Result.error("服务器正忙");
   }
}

# 过滤器

编写步骤：

  1. 定义类，实现 Filter接口，并重写其所有方法
  2. 配置Filter拦截资源的路径：在类上定义@WebFilter("/*") 注解
  3. 在doFilter方法中根据业务逻辑放行
// filterName表示过滤器名称    urlPatterns表示拦截路径
@WebFilter(filterName = "myFilter", urlPatterns = "/*")
@Slf4j
public class MyFilter implements Filter {
   @Override
   public void doFilter(ServletRequest servletRequest, ServletResponse servletResponse, FilterChain filterChain) throws IOException, ServletException {
      HttpServletRequest request = (HttpServletRequest) servletRequest;
      HttpServletResponse response = (HttpServletResponse) servletResponse;
      log.info("前端请求: {}",request.getRequestURI());
      filterChain.doFilter(request,response);	//	放行
   }

   @Override
   public void init(FilterConfig filterConfig) throws ServletException {
      Filter.super.init(filterConfig);
   }

   @Override
   public void destroy() {
      Filter.super.destroy();
   }
}

# 拦截器

拦截器：一种拦截处理器方法（Controller方法）的机制，可以在处理器方法执行前 / 后执行，底层采用的是AOP。 编写步骤：

  1. 编写拦截器，实现HandlerInterceptor接口

    需求：如果用户在请求指定地址的时候，如果session中没有user属性，则拦截请求，有则放行。

    @Component
public class LoginInterceptor implements HandlerInterceptor {
	@Override
	public boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler) throws Exception {
		System.out.println("登录拦截器");
		HttpSession session = request.getSession();
		Object user = session.getAttribute("user");
		response.setContentType("application/json;charset=utf-8");	//	解决响应中文乱码
		if (user == null) {
			// 方法1:手动返回请求结果
			response.getWriter().write("用户还没有登录，没有权限访问");
			// 方法2:抛出一个自定义异常(被全局异常处理器捕获)
			throw new CustomerException("用户未登录");
            return false;
		}
		return true;    //  是否放行
	}
}

# 拦截器和过滤器的区别 (opens new window)

  • 归属不同： Filter属于Servlet技术，Interceptor属于SpringMVC技术
  • 执行顺序不同：先执行过滤器，后执行拦截器
  • 拦截内容不同： Filter是拦截请求和响应的，Interceptor 拦截访问处理器中的方法，也就是拦截Controller中的方法 调用。
  • 实现原理不同：Filter 是基于函数回调的，Interceptor 则是基于Java的反射机制（动态代理）实现的
  • 拦截的请求范围不同：Filter几乎可以对所有进入容器的请求起作用，而Interceptor只会对Controller中请求或访问static目录下的资源请求起作用。

# SpringMVC的执行流程

DispatcherServlet 前端控制器 作用：核心对象，调度工作流

Spring MVC 的三大组件

  • 处理器映射器 HandlerMapping：解析Controller中的路径，和我们客户端的路径进行匹配，找到要访问的Controller中的方法
  • 处理器适配器 HandlerAdapter：负责去调用Controller中的对应的方法，执行该方法，将结果封装成ModelAndView对象。
  • 试图解析器 ViewResolver：负责解析ModelAndView对象中的数据和页面，最终告诉前端控制器去响应。
