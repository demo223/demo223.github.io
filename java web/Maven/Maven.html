# Maven

# Maven介绍

Maven是专门用于管理和构建Java项目的工具，它的主要功能有：

  • 提供了一套标准化的项目结构

  • 提供了一套标准化的构建流程（编译，测试，打包，发布……）

  • 提供了一套依赖管理机制

  1. # 标准化的项目结构

    项目结构我们都知道，每一个开发工具（IDE）都有自己不同的项目结构，它们互相之间不通用。我在eclipse中创建的目录，无法在idea中进行使用，这就造成了很大的不方便。

    而Maven提供了一套标准化的项目结构，所有的IDE使用Maven构建的项目完全一样，所以IDE创建的Maven项目可以通用。

# Maven仓库

# Maven仓库分类

  • 本地仓库：本机上面设置的Maven仓库地址
  • 中央仓库 (opens new window)：由Maven团队维护的全球唯一的仓库
  • 远程仓库(私服)：一般由公司团队搭建的私有仓库(如阿里云镜像)

当项目中使用坐标引入对应依赖jar包后，首先会查找本地仓库中是否有对应的jar包：

  • 如果有，则在项目直接引用;
  • 如果有远程仓库，则接着去远程仓库下载
  • 如果没有，则去中央仓库中下载对应的jar包到本地仓库。

jar包的查找过程是一个由近及远的过程 本地仓库 --> 远程仓库--> 中央仓库

# Maven仓库的本地安装配置

  1. 去官网下载apache-maven-x.x.x.zip的压缩包并加压到本地，注意文件路径不要带中文。

    • bin目录 ： 存放的是可执行命令。mvn 命令重点关注。
    • conf目录 ：存放Maven的配置文件。settings.xml 配置文件后期需要修改。
    • lib目录 ：存放Maven依赖的jar包。Maven也是使用java开发的，所以它也依赖其他的jar包。

  2. 配置Maven的环境变量

    在环境变量的系统变量中添加一个MAVEN_HOME的系统变量，检查系统变量里面是否已经配置过JAVA_HOME了，这两个系统变量一定要配置。

    在系统变量的path变量里面配置Maven的bin文件路径

    打开命令终端进行验证，输入mvn -version

    出现上图界面即表示安装成功

  3. 配置本地仓库(设置settings.xml)

    • 修改文件conf/settings.xml 中的 <localRepository> 指定目录作为本地仓库，用来存储jar包。

    • 配置阿里云镜像

      中央仓库在国外，所以下载jar包速度可能比较慢，而阿里公司提供了一个远程仓库，里面基本也都有开源项目的jar包。修改 conf/settings.xml 中的 <mirrors>标签

    • 配置 默认1.8jdk

    settings.xml文件中的配置信息如下

    <?xml version="1.0" encoding="UTF-8"?>

<settings xmlns="http://maven.apache.org/SETTINGS/1.0.0"
          xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
          xsi:schemaLocation="http://maven.apache.org/SETTINGS/1.0.0 http://maven.apache.org/xsd/settings-1.0.0.xsd">

    <!-- 1. 配置本地仓库目录 -->
    <localRepository>d:/software/maven_repository</localRepository>

    <pluginGroups></pluginGroups>
    <proxies></proxies>
    <servers></servers>

    <mirrors>
        <!-- 2. 配置阿里云镜像仓库 -->
        <mirror>
            <id>alimaven</id>
            <name>aliyun maven</name>
            <url>http://maven.aliyun.com/nexus/content/groups/public/</url>
            <mirrorOf>central</mirrorOf>
        </mirror>
    </mirrors>

    <profiles>
        <!-- 3. 配置默认jdk1.8 -->
        <profile>
            <id>jdk-1.8</id>
            <activation>
                <activeByDefault>true</activeByDefault>
                <jdk>1.8</jdk>
            </activation>
            <properties>
                <maven.compiler.source>1.8</maven.compiler.source>
                <maven.compiler.target>1.8</maven.compiler.target>
                <maven.compiler.compilerVersion>1.8</maven.compiler.compilerVersion>
                <maven.compiler.encoding>utf-8</maven.compiler.encoding>
                <project.build.sourceEncoding>UTF-8</project.build.sourceEncoding>
            </properties>
        </profile>
    </profiles>

</settings>

# Maven的基本使用

# Maven的常用命令

  • compile ：编译

  • clean：清理

  • test：测试

  • package：打包

  • install：安装

  • 执行mvn compile，如果本地Maven仓库中不存在，会自动从阿里云下载需要的jar包，查看本地仓库可以看到下载好的jar包。

    同时在项目下面会生成一个target文件夹，编译后的字节码文件就放在该目录下

  • 执行mvn clean

    • 从阿里云下载清理需要的插件jar包
    • 然后删除项目下的 target 目录

  • 执行mvn package

    • 从阿里云下载打包需要的插件jar包
    • 在项目的 terget 目录下有一个jar包（将当前项目打成的jar包）

  • 执行mvn test

    • 该命令会执行所有的测试代码

  • 执行mvn install

    • 该命令会将当前项目打成jar包，并安装到本地仓库。

# Maven的生命周期

Maven 对项目构建的生命周期划分为3套：

  • clean ：清理工作。
  • default ：核心工作，例如编译，测试，打包，安装等。
  • site ： 产生报告，发布站点等。这套声明周期一般不会使用。

同一套生命周期内，执行后边的命令，前面的所有命令会自动执行。例如默认（default）生命周期如下：

  • 当我们执行 install（安装）命令时，它会先执行 compile命令，再执行 test 命令，再执行 package 命令，最后执行 install 命令。

  • 当我们执行 package （打包）命令时，它会先执行 compile 命令，再执行 test 命令，最后执行 package 命令。

  • 生命周期的命令很，但是常用的一部分

# Maven项目依赖管理

# 添加依赖

在项目的pom.xml中编写<dependencies>标签，然后在改标签中加入<dependency>

在<dependency>标签中存放jar包的坐标，坐标一般包括常见的3个部分：

坐标的 groupId，artifactId，version简写为gav

具体的某一个jar包的坐标可以去如下网站进行搜索

https://mvnrepository.com/

上面的步骤在实际开发的时候是比较麻烦的，下面介绍其他更加简便的方法导入jar包的坐标

在 pom.xml 文件中 按下 alt + insert，选择 Dependency(或添加依赖性)

如果坐标在引入后没有生效，可以刷新一下Maven

当然也可以设置idea自动导入jar包

# 依赖范围

通过设置坐标的依赖范围(<scope>)，可以设置 对应jar包的作用范围：编译环境、测试环境、运行环境。

<dependency>
    <groupId>junit</groupId>
    <artifactId>junit</artifactId>
    <version>4.13</version>
    <scope>test</scope>
</dependency>

上面的代码中junit依赖的scope的范围是test，说明它只能在测试环境中使用

scope的取值有如下范围

依赖范围 编译环境 测试环境 运行环境 例子
compile ✅ ✅ ✅ logback
test ❌ ✅ ❌ Junit
provided ✅ ✅ ❌ servlet-api
runtime ❌ ✅ ✅ jdbc驱动
system ✅ ✅ ❌ 存储在本地的jar包
  • compile ：作用于编译环境、测试环境、运行环境。
  • test ： 作用于测试环境。典型的就是Junit坐标，以后使用Junit时，都会将scope指定为该值
  • provided ：作用于编译环境、测试环境。我们后面会学习 servlet-api ，在使用它时，必须将 scope 设置为该值，不然运行时就会报错
  • runtime ： 作用于测试环境、运行环境。jdbc驱动一般将 scope 设置为该值，当然不设置也没有任何问题

如果没有设置scope，默认是compile，对于大部分的jar包都是默认不设置scope的
